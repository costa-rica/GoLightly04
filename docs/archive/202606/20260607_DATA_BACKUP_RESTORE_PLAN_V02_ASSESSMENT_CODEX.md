# Assessment: Data Backup Restore Plan V02

## Qualifying Concerns

### 1. Archive safety is applied after extraction, but extraction itself still writes untrusted paths

Plan V02 adds `safeRestoreResources` with path containment, manifest validation,
regular-file checks, and symlink skipping. That addresses the resource-copy
portion of the V01 concern, but the restore flow still says to extract the
uploaded zip to `tempDir` first and only then inspect `tempDir/manifest.json` and
`tempDir/resources`.

The current route uses `unzipper.Extract({ path: tempDir })`, and V02 carries
that extraction shape forward. This means the implementation still performs a
filesystem write for every archive entry before the new containment rules run.
The post-extraction helper cannot protect against unsafe zip entries that are
written during extraction itself, and it also cannot reliably test source-path
containment after the archive library has already decided where each entry
lands.

This matters because the feature expands restore from a small DB-only zip to a
large arbitrary archive that may contain a resource tree. Even for an admin-only
endpoint, the plan should not depend on all uploaded archives being generated by
this app.

Recommended plan correction:

- Replace whole-archive `unzipper.Extract({ path: tempDir })` with an explicit
  safe extraction helper, or make `safeRestoreResources` stream entries directly
  from the zip.
- For each zip entry, normalize entry names to POSIX-style paths and reject
  absolute paths, empty paths, drive-prefixed paths, and any path containing
  `..` segments before writing to disk.
- Extract only the expected root-level CSV files, optional `manifest.json`, and
  allowed `resources/**` regular-file entries.
- Preserve the existing restore contract: legacy DB-only zips without a
  manifest still restore DB CSVs, while malformed unsafe entries are skipped or
  rejected before any write outside the intended temp tree can occur.
