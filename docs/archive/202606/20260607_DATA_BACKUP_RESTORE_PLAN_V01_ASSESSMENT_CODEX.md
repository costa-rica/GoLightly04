# Assessment: Data Backup Restore Plan V01

## Qualifying Concerns

### 1. API can report a backup as queued when no worker job was accepted

The plan instructs `POST /database/create-backup` to call `requestWorkerBackup`
and return HTTP 202 immediately, while the proposed `requestWorkerBackup`
retries and then logs without throwing after all attempts fail
(`docs/20260607_DATA_BACKUP_RESTORE_PLAN_V01.md`, worker client section).

That creates a correctness problem for this feature. The requirements explicitly
avoid a job-status/history system; the completed file appearing in the admin
list is the only success signal. If the worker is down, misconfigured, or
returns `409` because a backup is already running, the browser would still show
"Backup job queued" even though no backup will appear. The existing
`notifyWorker` pattern is not a perfect precedent because meditation processing
has database state to reconcile later; this backup flow has no persistent job
record or retry worker.

Recommended plan correction:

- Treat the API-to-worker call as synchronous only until worker acceptance.
- If the worker returns `202`, then the API returns `202`.
- If the worker returns `409`, propagate a conflict to the admin UI.
- If the worker is unreachable after retries, return a non-2xx API error so the
  admin is not told a backup was queued.

### 2. Restore resource handling needs explicit archive safety rules

The plan extends `POST /database/replenish-database` to extract an uploaded zip,
read `manifest.json`, and copy `tempDir/resources/**` into
`PATH_PROJECT_RESOURCES`, overwriting conflicts. The plan does not specify
path-containment, symlink, or manifest validation rules for the extracted
archive.

This matters because restore now writes a large arbitrary directory tree to the
server filesystem. Even though the route is admin-only, the implementation
should not rely on all uploaded archives being generated by this app. Without
explicit rules, an implementer may recursively copy symlinks, follow paths that
escape `resources/`, or restore files into excluded backup directories if a
crafted archive contains those entries.

Recommended plan correction:

- Validate manifest shape before enabling resource restore.
- Only restore regular files below the extracted `resources/` root.
- Resolve and check both source and destination paths before copy; skip or fail
  if a path escapes `tempDir/resources` or `PATH_PROJECT_RESOURCES`.
- Do not follow symlinks during restore.
- Preserve the backup-directory exclusion on restore as well, or explicitly
  document that uploaded `resources/backups_db*` entries are rejected.

### 3. Backup resource walking should define symlink behavior

The worker backup service is instructed to recursively walk
`PATH_PROJECT_RESOURCES`, skip `backups_db/` and `backups_db_and_data/`, and
copy each file into the staging directory. The repo currently stores expected
resource directories as normal local folders, but the plan does not say whether
to follow symlinks.

If a production resources folder ever contains a symlink, following it could
copy data outside `PATH_PROJECT_RESOURCES`, recurse unexpectedly, or produce a
backup that cannot be safely restored. This is a feasibility and correctness
risk for a backup feature whose purpose is a portable environment snapshot.

Recommended plan correction:

- Use `lstat` during the backup walk.
- Copy regular files only.
- Skip symlinks and non-regular filesystem entries, logging what was skipped.
