# Create Database Docs

This skill instructs AI coding agents how to generate a standard documentation set for the database layer of a monorepo that contains a `db-models/` package. Run it once per project (or whenever the underlying schema and infrastructure have drifted far enough from the docs that a regeneration is cheaper than incremental edits).

## When to use

- The repository has a `db-models/` (or equivalent shared ORM) package and no existing `docs/db-models/` directory.
- The existing `docs/db-models/` is stale, fragmented across many dated runbooks, or missing one of the four canonical files.
- The user explicitly asks to "create the database docs", "regenerate db-models docs", or similar.

## Output location and files

All output files MUST be written to `docs/db-models/` at the repository root. Produce exactly these four files (do not invent extras, do not skip any):

1. `docs/db-models/TABLE_REFERENCE.md`
2. `docs/db-models/USAGE_GUIDE.md`
3. `docs/db-models/SETUP_MAC.md`
4. `docs/db-models/SETUP_UBUNTU.md`

Before writing, read the project's existing AGENTS.md / CLAUDE.md, the `db-models/` source (especially `src/models/*.ts`, `src/models/_connection.ts`, `src/models/_associations.ts`, `src/models/_index.ts`), and any setup or drop/restore runbooks already in the repo. Source content from the codebase, not from training data — model names, column types, env var names, and role names must match what is actually defined in the project.

## Section conventions

Every generated file uses GitHub-flavored Markdown with the YAML frontmatter described below. Within each file, use `##` for top-level sections and `###` for subsections. Keep prose tight and prefer tables or fenced code blocks over paragraphs when the content is structured.

## 1. YAML frontmatter (every file)

Every generated `.md` file — including this one and the four output files — MUST begin with a YAML frontmatter block. The frontmatter block is delimited by `---` lines and contains exactly these four keys:

```yaml
---
created_at: YYYY-MM-DD
updated_at: YYYY-MM-DD
created_by: <agent name> (<model>)
modified_by: <agent name> (<model>)
---
```

Rules:

- `created_at` is set once, at file creation, and MUST NEVER be modified on later edits.
- `updated_at` is rewritten to today's date on every modification, even if `created_at` and `updated_at` were identical before.
- `created_by` is set once, at file creation, and MUST NEVER be modified on later edits.
- `modified_by` is rewritten on every modification to reflect the agent that performed the change. On the very first write, set it to the same value as `created_by`.
- The `created_by` / `modified_by` value uses the format `<agent name> (<model>)`, lowercase only, with no email addresses and no angle brackets.

Acceptable examples:

```yaml
created_by: claude (sonnet-4)
created_by: claude (opus-4.7)
created_by: codex (gpt-5)
modified_by: claude (haiku-4.5)
```

Not acceptable:

- `created_by: Claude <noreply@anthropic.com>` (contains email + angle brackets + capitalization)
- `created_by: claude` (missing model)
- `created_by: Claude (Sonnet-4)` (not lowercase)

When editing an existing file that already has frontmatter, update only `updated_at` and `modified_by`. When editing a file that lacks frontmatter, add it and set `created_at` to today's date, `created_by` to the editing agent (the historical author is unknown and should not be guessed).

## 2. TABLE_REFERENCE.md

Purpose: a complete, browsable reference of every table in the database, generated by reading the model source files.

Required structure:

1. Frontmatter (per §1).
2. `# Table Reference` heading and a one-sentence description that names the project, the ORM, and the database engine (read these from `db-models/`).
3. A note stating that all tables include `createdAt` and `updatedAt` timestamp fields if the project uses Sequelize defaults — otherwise document the actual timestamp convention.
4. `## Core Tables` — every model that represents a primary entity.
5. `## Junction Tables` — every model that exists only to join two other tables (typical names: `*Contract`, `*Join`, `*Through`).

For each table, produce a `###` subsection with:

- `### <ModelName>` heading (use the TypeScript class name, not the SQL table name).
- A bullet for `**Table name**: ` `` `<sqlTableName>` `` (the Sequelize `tableName` option or the auto-pluralized default).
- A Markdown column table with headers `Column | Type | Constraints | Notes`. Read columns directly from the model's `init()` call. Mark primary keys as `PK, Auto Increment`, foreign keys as `FK, NOT NULL` or `FK, NULL`, defaults as `DEFAULT <value>`. In the Notes column, write the FK target as `→ <TargetTable>.id` and add short prose for non-obvious columns.
- An `**Indexes**:` bullet list if the model declares non-default indexes (read from the `indexes` option in `init()`).
- A `**Relationships**:` bullet list naming every `belongsTo`, `hasOne`, `hasMany`, and `belongsToMany` declared in `_associations.ts`. Use the pattern `belongsTo <Target> (via \`<fkColumn>\`)`. For `belongsToMany`, include the `through` table.
- A `---` horizontal rule separating tables.

Do not invent columns or relationships. If a column's purpose is non-obvious and cannot be inferred from the code, leave the Notes cell empty rather than guessing.

## 3. USAGE_GUIDE.md

Purpose: explain how a consuming app in the monorepo imports and uses the `db-models/` package.

Required structure:

1. Frontmatter (per §1).
2. `# Database Usage Guide` heading and a one-sentence summary identifying the package by its published name (e.g. `@<scope>/db-models`).
3. `## Package Description` — bulleted facts: published name, source path, build output path, model convention (one class per table, centralized init, centralized associations), whether timestamps are enabled by default.
4. `## Project Structure` — a fenced `text` block showing the layout of `db-models/src/` (connection file, index, associations file, individual model files). Generate this by reading the actual directory, not from memory.
5. `## Using This Package in an App` — describe the required startup sequence as a numbered list (load env, import init function and sequelize handle, call `initModels()`, authenticate, sync, then mount routes / start server). Follow with a `### Initialization pattern` fenced code block showing the minimal bootstrap function. Use the project's actual import path and function names.
6. `## Why Order Matters` — short bullets explaining what breaks if the steps are reordered (model not initialized, table missing, connection not ready).
7. `## Environment Variables` — list every env var read by `_connection.ts`. Mark required vs. optional, give an example value for each, and document any filesystem or network preconditions (directory must exist, port must be reachable, role must have CONNECT, etc.).
8. `## Creating or Updating Schema` — a fenced code block showing the three `sequelize.sync()` modes (plain, `{ alter: true }`, `{ force: true }`) with a one-line note on when each is appropriate and which is destructive.
9. `## Using Models` — a small fenced code block showing two or three realistic queries with actual model names from the project.
10. `## Template (copy for each new model)` — a TypeScript fenced block showing the canonical pattern for adding a new model: imports, class with `InferAttributes` / `InferCreationAttributes`, `init<ModelName>()` function, `tableName`, `timestamps`. Match the project's existing model files exactly.
11. `## Example src/models/_index.ts` — a fenced TypeScript block showing how a new model is wired into the central init function and re-exported. Mirror the format used in the project's current `_index.ts`.
12. `## Database Configuration` — closing bullet list: engine name, env var prefix, and whether the package itself owns a `.env` or reads from the consuming app.

## 4. SETUP_MAC.md

Purpose: stand up the database on a developer's Mac workstation from scratch, and recover it from a backup when needed.

Required structure:

1. Frontmatter (per §1).
2. `# Local Database Setup — Mac` heading and one sentence stating the workstation target (Homebrew-managed engine on macOS).
3. `## First Build` — the from-zero install flow. Subsections (`###`) in this order:
   - `### 1. Install the database engine` — Homebrew install command, service start command, PATH export for the engine's bin directory.
   - `### 2. Create roles and databases` — create the bootstrap role (with `CREATEDB`) and the runtime app role; create the dev database owned by the bootstrap role; create one test database per package that runs tests (read package list from the repo).
   - `### 3. Grant schema privileges` — grant `CREATE` on `public` to the bootstrap role, `USAGE` to the app role, and set `ALTER DEFAULT PRIVILEGES` so future tables and sequences are accessible to the app role.
   - `### 4. Seed the dev database` — invoke the project's database manager / restore tool to import a backup zip (or, if no backup exists, sync an empty schema). Use the project's actual tool name and flag names.
   - `### 5. Wire up each package's .env` — for each package that talks to the database, a fenced block showing the env keys (host, port, database, user, password, schema, plus any pool/SSL tuning). Mark which packages use the bootstrap role vs. the app role.
   - `### 6. Sanity checks` — commands to confirm the engine is up, the role can log in, and at least one package can boot against the database.
4. `## Restore` — the drop-and-restore flow when the dev database is corrupted, drifted, or needs a refresh from a known-good backup. Subsections (`###`) in this order:
   - `### 1. Verify the engine is running` — readiness check and how to start the service if it is not.
   - `### 2. Confirm database, roles, and ownership` — short queries to print the database list, the project's roles, and the dev database owner.
   - `### 3. Stop local services` — list every long-running dev process in the monorepo (api, portal, workers, model watcher) and how to stop them; include a port check (`lsof`) for the project's known dev ports.
   - `### 4. Confirm no active connections` — query `pg_stat_activity` (or the engine equivalent) for the dev database and provide a terminate-backends command for stragglers.
   - `### 5. Optional backup before dropping` — run the project's backup tool, point at the project's log file for progress.
   - `### 6. Drop the database` — single-line drop command plus a confirmation query.
   - `### 7. Recreate the empty database and reapply privileges` — create the empty database with the bootstrap role as owner, then re-grant all the privileges from First Build §3 (these are wiped by the drop). Include a verification query that prints schema privileges for both roles.
   - `### 8. Restore from a backup` — invoke the project's restore tool with the backup zip path. Note any tool-specific flag spelling (e.g. `--zip_file` vs `--zip-file`) and remind agents to follow the log file.
   - `### 9. Verify the restored database` — count tables, run a focused build on each consuming package.
   - `### 10. Restart local services` — list the dev commands for each package in startup order.
5. `## Quick Reference` — a closing table with: engine install method, host, port, dev database name, owner role, runtime role, schema name, and the database manager log path.

## 5. SETUP_UBUNTU.md

Purpose: stand up the database on the production Ubuntu host and recover it from a backup when needed.

Required structure:

1. Frontmatter (per §1).
2. `# Production Database Setup — Ubuntu` heading and one sentence stating the production target (apt-managed engine on Ubuntu, sitting behind the project's reverse proxy or service manager).
3. `## First Build` — the from-zero install flow on a fresh server. Subsections (`###`) in this order:
   - `### 1. Verify the engine is running` — readiness check and `systemctl status`.
   - `### 2. Allow local connections without a password` — edit `pg_hba.conf` (or the engine equivalent) to use `trust` for `127.0.0.1/32`, reload the service, and confirm the port is not open to the public network (`ufw status`).
   - `### 3. Create roles` — create the bootstrap role (with `CREATEDB`) and the runtime app role using `sudo -u <engine_superuser>`.
   - `### 4. Create the production database` — create the prod database owned by the bootstrap role; grant `CONNECT` to the app role.
   - `### 5. Grant schema privileges` — grant `CREATE` on `public` to the bootstrap role, `USAGE` to the app role, and set `ALTER DEFAULT PRIVILEGES` so future tables and sequences are accessible to the app role.
   - `### 6. Configure each package's .env` — for each package, a fenced block showing the env keys. Leave `PG_PASSWORD` (or equivalent) blank because the `trust` line in step 2 removes the need. Mark which packages use the bootstrap role vs. the app role.
   - `### 7. Load data with the database manager` — invoke the project's restore tool with a backup zip; also document the dry-run variant if the tool supports one.
   - `### 8. Verify the connection` — a single representative query (e.g. `SELECT COUNT(*) FROM "<MainTable>";`) using the app role.
4. `## Restore` — the drop-and-restore flow on the production host. Subsections (`###`) in this order:
   - `### 1. Stop all services` — `systemctl stop` lines for every project service unit; finish with a status check.
   - `### 2. Confirm no active connections` — `pg_stat_activity` query and a terminate-backends statement.
   - `### 3. Drop the database` — `dropdb` command; confirm with `psql -l`.
   - `### 4. Recreate the empty database and reapply privileges` — `createdb` plus the schema-grant block. Call out that PG 15+ revokes `CREATE` on `public` by default and that the recreate step will fail without re-granting it.
   - `### 5. Stage the backup file` — if the project uses a `limited_user` or other restricted account for the restore, document the `cp` from the operator's home directory to the restricted account's home; otherwise omit this subsection.
   - `### 6. Restore from the backup zip` — invoke the project's restore tool, optionally via `nohup` if the run is long. Include the log-follow command.
   - `### 7. Restart services` — `systemctl start` lines for every project service unit; finish with a status check.
5. `## Quick Reference` — a closing table with: prod database name, owner role, app role, port, host, and log file path.

## Checklist before finishing

Before reporting the task complete, verify each item:

- [ ] All four files exist under `docs/db-models/`.
- [ ] Every file starts with the four-key YAML frontmatter block.
- [ ] `created_at` and `created_by` are set on each file; `updated_at` matches today; `modified_by` matches the agent doing the writing.
- [ ] Model names, column names, env vars, role names, service names, and tool flag spellings were read from the project's source — none were invented.
- [ ] `TABLE_REFERENCE.md` covers every model in `db-models/src/models/` and groups them into Core vs. Junction.
- [ ] `USAGE_GUIDE.md`'s code samples use the project's actual import paths and exported names.
- [ ] `SETUP_MAC.md` and `SETUP_UBUNTU.md` each contain a `## First Build` and a `## Restore` section, in that order.
