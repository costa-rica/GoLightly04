## Phase 8 — Meditations router (`/meditations`)

**Canonical element DTO** (per V04): incoming `meditationArray` is an array of `{ id, text?, voice_id?, speed?, pause_duration?, sound_file? }`. Type is derived server-side from which optional field is populated (`text` → `'text'`, `pause_duration` → `'pause'`, `sound_file` → `'sound'`). Import the type from `@golightly/shared-types`.

- [x] `POST /meditations/create` — authed; intake handler that:
  - Derives `type` per element from the populated field (text / pause_duration / sound_file)
  - Creates `meditations` row (`status: 'pending'`, `visibility` from body defaulting to `'public'`, `meditation_array` JSONB = exact payload plus a computed `sequence` per element)
  - For each element creates a `jobs_queue` row with `type`, `sequence`, `input_data` (JSON string)
    - `text`: `input_data = JSON.stringify({ text, voice_id?, speed? })`
    - `sound`: `input_data = JSON.stringify({ sound_file })`
    - `pause`: `input_data = JSON.stringify({ pause_duration })`
    - initialize recovery/audit fields: `attempt_count = 0`, `last_error = null`, `last_attempted_at = null`
  - `sound` rows are immediately marked `complete` with `file_path` resolved from `prerecorded_audio/`
  - `pause` rows are marked `complete` immediately (no file)
  - `text` rows stay `pending` for the worker
  - POSTs `{ meditationId }` to `URL_WORKER_NODE/process` via the worker client (Phase 12)
  - Responds `{ message, queueId, filePath }` (use the meditation's id as queueId per existing frontend contract)
- [x] `GET /meditations/all` — optional auth; returns public + user's private meditations. Joins `contract_user_meditations` to populate `isFavorite` per meditation when authed. Sets `isOwned` and `ownerUserId`.
- [x] `GET /meditations/:id` — optional auth; single meditation detail (new endpoint)
- [x] `GET /meditations/:id/stream` — accepts either `Authorization: Bearer <jwt>` **or** `?token=<signed-stream-token>` query param (see below); streams the final MP3 with `Range` header support; **auto-increments `listen_count`** once per request (only when the request has no `Range` header or has `Range: bytes=0-`, to avoid double-counting range continuations)
- [x] `GET /meditations/:id/stream-token` — authed; issues a short-lived signed stream token (~5 min) usable as `?token=` on the stream endpoint. Token payload: `{ meditationId, userId, exp }`. Signed with `JWT_SECRET`. This enables browser `<audio>` native streaming without the blob-fetch workaround in the old web app.
- [x] `POST /meditations/favorite/:meditationId/:trueOrFalse` — authed; insert/delete row in `contract_user_meditations`
- [x] `PATCH /meditations/update/:id` — authed, owner-only; update title/description/visibility
- [x] `DELETE /meditations/:id` — authed, owner-or-admin; calls `deleteMeditationCascade(meditationId)` service
- [x] Tests for each route

## Phase 9 — Cascade delete service

- [x] `src/services/meditations/deleteMeditationCascade.ts` — single function that performs full cascade delete for a meditation:
  1. Delete `jobs_queue` rows for the meditation
  2. Delete `contract_user_meditations` rows for the meditation
  3. Delete the `meditations` row
  4. Delete on disk: all files matching `eleven_labs_audio_files/*/el_{meditationId}_*.mp3`
  5. Delete on disk: `meditation_soundfiles/*/meditation_{meditationId}.mp3`
  6. **Never** touches `prerecorded_audio/`
- [x] Wrap DB deletes in a transaction; perform filesystem deletes after commit
- [x] Used by: `DELETE /meditations/:id`, `DELETE /admin/meditations/:id`, `DELETE /admin/queuer/:id`
- [ ] Tests: fixture a meditation with 2 text jobs + 1 sound job + 1 favorite row → run cascade → assert all gone; assert `prerecorded_audio/` untouched

- [x] `POST /admin/meditations/:id/requeue` — explicit retry path that preserves failure history:
  - does **not** reset failed jobs back to `pending`
  - validates the meditation exists and has at least one `jobs_queue` row with `status != 'complete'`, or that all jobs are `complete` but `meditations.status != 'complete'` and concatenation must be retried
  - may set `meditations.status` from `failed` to `pending` only as a meditation-level lifecycle reset before handing work back to the worker, but job-level failure history remains intact
  - sends `POST { meditationId, mode: 'requeue' }` to the worker
  - worker requeue mode is responsible for selecting retryable rows (`failed`, `pending`, and any stale `processing` rows normalized by reconciliation); the API does not rewrite job statuses to hide prior failures
  - responds `{ message, meditationId }`

  - assert requeue is allowed when all jobs are `complete` but concatenation must be retried
